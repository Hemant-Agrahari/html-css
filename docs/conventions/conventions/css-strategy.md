# CSS Strategy

How stylesheets are authored, bundled, purged, and loaded. Goal: one inline critical style
block under 14 KB gzipped + one blocking per-page stylesheet + everything else deferred.

## File taxonomy and naming

| Location | Example | Role |
|---|---|---|
| `shared/css/vendor/` | `site-main.css`, `site-components.css`, `fonts.css` | **Source vendor CSS** — never edited, never linked directly by built pages |
| `shared/css/global.css` | `global.css` | **Our overrides** — edited freely, joins the purge input |
| `shared/css/pages/{slug}.css` | `home.css`, `about.css`, `service.css` | **Per-page source** — edited freely, joins the purge input for pages that use it |
| `shared/css/pages/{slug}.purged.css` | `white-label-igaming-platform.purged.css` | **Purged output** — generated by `tools/purgecss-pages.js`, linked blocking in page |
| `shared/css/pages/{slug}.critical.css` | `white-label-igaming-platform.critical.css` | **Inlined critical output** — generated by `tools/extract-critical-css-page.js`, inlined in `<style>` |

Naming rules:
- `*.purged.css` → auto-generated, safe to delete + regenerate, **linked** in head
- `*.critical.css` → auto-generated, safe to delete + regenerate, **inlined** in head (not linked)
- Plain `.css` files → source, checked into git, hand-edited

## Loading pattern per page

Target pattern for every built page:

```html
<!-- preload fonts -->
<link rel="preload" as="font" type="font/woff2" href="/assets/fonts/outfit-latin.woff2" crossorigin>

<!-- inline critical, above-the-fold, <14 KB gzipped -->
<style>/* contents of shared/css/pages/{slug}.critical.css */</style>

<!-- single deferred sheet: purged page CSS (bootstrap classes actually used included) -->
<link rel="stylesheet" href="/css/pages/{slug}.purged.css" media="print" onload="this.media='all'">
<noscript><link rel="stylesheet" href="/css/pages/{slug}.purged.css"></noscript>
```

That is the only pattern. Variations must be justified in the page's frontmatter and logged here.

## What goes into the purge input

The per-page purge input is:

```
shared/assets/vendor/css/bootstrap.min.css
shared/css/vendor/site-main.css
shared/css/vendor/site-components.css
shared/css/vendor/site-pages.css
shared/css/vendor/site-templates.css
shared/css/vendor/site-service-pages.css
shared/css/vendor/fonts.css
shared/css/global.css
shared/css/pages/{slug}.css    (if it exists)
```

**Bootstrap belongs in the purge input.** Loading full Bootstrap separately is a ~150 KB blocking
cost that currently dominates LCP on mobile. Adding it to the PurgeCSS combiner means every page's
purged file contains only the ~8–12 KB of Bootstrap it actually uses.

## What stays deferred (never in the blocking path)

The purged CSS is the only deferred stylesheet per page, loaded via `media="print" onload` swap.
All vendor CSS libraries have been **removed entirely** (April 2026):

| Removed | Size | Replaced with |
|---|---|---|
| `slick.css` + `slick-theme.css` | ~10 KB | `.vc-carousel` rules in `global.css` (~200 bytes) |
| `aos.css` | ~6 KB | `.aos-item` rules in `global.css` (~200 bytes) |
| `fa-subset.css` + font files | ~303 KB | 7 inline SVGs in HTML (~1 KB total) |
| `fonts.css` | ~1.1 KB | Inlined with critical CSS by `build.js` |

**`head-meta.html` now has zero `<link rel="stylesheet">` tags.** All CSS is either inlined
(critical + fonts) or deferred (purged). This eliminates all deferred CSS network requests
except the single purged file per page.

**Scroll-reveal animations:** `.aos-item` starts `opacity: 0; transform: translateY(30px)` and
transitions to normal when `.aos-animate` is added by `IntersectionObserver` in `aos-init.js`.
Rule: **never add `.aos-item` to above-the-fold elements.** The transition starts from invisible.

## Safelist discipline

`tools/purgecss-pages.js` has a `SAFELIST` for classes JavaScript adds at runtime (`show`, `active`,
`collapsing`, etc.). Every time a JS module toggles a class that's not already in the safelist,
the class must be added — otherwise PurgeCSS strips it and the UI breaks silently.

Rule: when adding JS behavior that toggles a class, update the safelist in the same commit.

## Authoring rules

- **R05** — No inline `style="..."` in page HTML or components. Use a class. Inline styles can't be purged, can't be cached, can't be overridden cleanly, and bloat HTML (= slower FCP).
- **R04** — No `@import` in any CSS file. `@import` blocks parallel download. Use a `<link>` or combine at build time.
- **R06** — Exactly one `<link rel="stylesheet">` per built page (the purged.css with its `<noscript>` fallback). Anything else in the critical path is a bug.
- **No unprefixed experimental properties** that rely on a vendor-only shipped feature.
- **Critical CSS under 14 KB gzipped.** Past that you've paid the cost without the benefit (first TCP congestion window).

## Critical CSS extraction — pitfalls

Extracting critical CSS by "elements visible in the viewport" is **not enough**. The extractor
must also include CSS rules for elements that are **present in the DOM but hidden** at the test
viewport — especially inside `<header>` and `<nav>`.

### The mobile-menu trap (April 2026 incident)

Our first extractor used `getBoundingClientRect()` to decide if a rule was critical. This skipped
rules for `display: none` elements (zero rect). On mobile, `.main_menu` is `display: none` at
≤1024px — so its hide rule was never extracted. Result:

- First paint with critical CSS only → mobile menu rendered **fully expanded** (header 1176px tall)
- Deferred CSS arrives → menu collapses to hamburger (header 64px)
- **CLS = 0.9483** — catastrophic shift, far worse than blocking the full CSS would have been

### The rule

Critical CSS must include any rule whose selector matches an element **inside `<header>` or
`<nav>`**, even if that element has zero bounding rect (hidden). These rules are often *why*
the element is hidden; excluding them inverts the initial layout.

The extractor at `tools/extract-critical-css-page.js` implements this via an `isInNavChain()`
helper. If you modify the extraction logic, preserve this behavior.

### How to verify before shipping

See the **Critical-CSS self-test** in `docs/cwv-definition-of-done.md`. Any new page's critical
CSS must pass that gate: pre-deferred layout (all deferred CSS blocked) must match post-deferred
layout. A difference means critical CSS is incomplete.

## Build order for CSS changes (MANDATORY — violating this caused a live LCP regression)

When modifying any CSS source (vendor rewrite, new background images, format conversions),
the build pipeline MUST run in this exact order:

```
1. Edit vendor CSS sources       (shared/css/vendor/, shared/css/global.css)
2. node tools/rewrite-bg-css.js  (updates vendor .png/.jpg → .webp)
3. node build.js                 (generates dist/ from sources)
4. node tools/purgecss-pages.js  (regenerates purged CSS from vendor+global)
5. cp dist/css/pages/*.purged.css shared/css/pages/
6. Rewrite purged + critical CSS (catch stale .png/.jpg refs — see below)
7. node tools/extract-critical-css-page.js --all
8. node build.js                 (final build with updated critical CSS)
9. VERIFY: grep -rE 'background-image.*\.(png|jpg)' dist/*.html
   → must return zero matches (except it-revolution-bg.png which has no smaller WebP)
```

**Step 6 is critical.** The `rewrite-bg-css.js` tool only updates vendor source CSS.
But `shared/css/pages/*.purged.css` and `*.critical.css` are generated artifacts that may
contain stale PNG/JPG refs from a previous extraction. They must also be rewritten.

### Why this matters (April 2026 incident)

After converting CSS backgrounds to WebP, the vendor CSS was correct (.webp refs) but
52 purged/critical CSS files still referenced the old `.jpg`/`.png` URLs. The browser
inlined those stale refs in `<style>`, downloaded the 39 KB JPEG instead of the 3 KB WebP,
and LCP regressed from 1.7 s back to 3.9 s on live. PSI score dropped from 97 to 82.

Fix was a direct rewrite of all purged + critical CSS files. But the root cause was
**skipping step 6** — the purged/critical files were committed with stale refs.

## Minification (April 2026)

CSS minification runs inside `build.js` at build time — no separate step needed.

### What is minified

| Target | Tool | When |
|---|---|---|
| All `dist/css/**/*.css` (purged + page CSS) | CleanCSS level 2 | End of `node build.js` (full build only, not `--page=`) |
| Inline critical CSS (`<style>` block) | CleanCSS level 2 | Per-page, during critical CSS injection |

**Level 2** means aggressive restructuring: shorthand merging, redundant rule removal, `0px → 0`,
colour shortening, duplicate property stripping. This is safe on purged CSS because all variable
references have already been resolved by the time CleanCSS runs.

**Source files in `shared/` are never minified.** Only `dist/` output is touched. Re-running
`node build.js` regenerates minified output; the sources stay readable.

### File size expectations

After the April 2026 CleanCSS integration:

| Metric | Before | After |
|---|---|---|
| Total `dist/css/` size | 20,085 KB | 3,853 KB (-81%) |
| `white-label-igaming-platform.purged.css` | 926 KB | 116 KB (-87%) |
| Inline critical CSS block (white-label) | ~63 KB | ~53 KB (-16%) |

The white-label page had the largest purged file because the critical CSS extractor ran dual-viewport
passes that duplicated Bootstrap grid blocks (see below). CleanCSS removed redundant rules
post-extraction but couldn't deduplicate structurally identical `@media` blocks.

### Critical CSS duplication trap

The critical CSS extractor (`tools/extract-critical-css-page.js`) runs two Playwright passes per page —
one at mobile (375×812) and one at desktop (1440×900). It deduplicates by exact string match only.
Bootstrap `@media` blocks from each pass have different content (the breakpoint filter activates
different rules at each viewport), so deduplication misses them and Bootstrap's grid ends up
included twice.

**Symptom:** critical CSS for pages with a large hero section (`franchise-banner`, `main-banner`)
is 50–70 KB instead of the expected 20–30 KB.

**Why it also inflates:** the `isInHero` flag in the extractor includes ALL CSS rules for ANY
element inside the hero section, even deeply nested elements below the visible fold within that
section. Combined with the dual-viewport Bootstrap duplication, this inflates the inline `<style>`
block significantly.

**Mitigation:** CleanCSS level 2 applied after extraction removes redundant rules within each block.
The inline `<style>` block stays under the 14 KB gzipped target for most pages.

**Future fix:** re-run `node tools/extract-critical-css-page.js --all` after any major PurgeCSS
rebuild — the smaller purge input feeds a smaller critical output automatically.

### PurgeCSS output is unminified

PurgeCSS strips unused selectors but emits expanded multiline CSS (including re-expanding
`bootstrap.min.css`). This is intentional — CleanCSS runs afterward in `build.js` and handles
minification. Do not pipe PurgeCSS output through a separate minifier first; CleanCSS level 2
is more aggressive and handles both in a single pass.

## Font display strategy (April 2026)

`fonts.css` uses `font-display: optional` (not `swap`).

**Why:** `font-display: swap` caused measurable CLS (0.616 on white-label page, PSI desktop)
because the fallback → Outfit swap shifted text layout. With `optional`, the browser uses Outfit
if it's already cached (return visitors) or downloads it in the background for next navigation.
First-time visitors on slow connections see Arial instead of Outfit for that single page load —
acceptable trade-off for eliminating font-related CLS site-wide.

**Evidence:** PageSpeed Insights desktop audit, April 2026 — CLS 0.616 with `swap`.

The `Outfit Fallback` @font-face (Arial with `size-adjust: 99.82%`, `ascent-override: 100.18%`)
remains in place. With `optional`, the fallback metrics still matter: if the browser decides not
to use Outfit (first visit, slow network), the fallback font renders with near-identical metrics
so the visual difference is minimal.
